## 2. Architecture Review

### 2.1 잘 된 점

**Protocol 기반 DI 경계가 명확합니다.**
- `UniverseProvider`
- `MarketDataProvider`
- `IndicatorEngine`
- `CandidateScorer`

이 경계 덕분에 pipeline 테스트가 fake/stub 중심으로 구성될 수 있고, 외부 API와 점수 로직이 강하게 엉키지 않습니다.

**failure isolation이 ticker 단위로 작동합니다.**
특정 종목 fetch 또는 계산 실패가 run 전체를 깨지 않고 `data_failures`에 축적됩니다. 외부 provider에 의존하는 batch screener로는 올바른 선택입니다.

**staged intraday merge 설계가 깔끔합니다.**
`PreferredIntradaySnapshotMarketDataProvider`가 base provider를 감싸 same-day 마지막 bar만 교체합니다. 기존 daily provider 경로를 깨지 않고 보강 데이터를 붙이는 방식이라 구조가 자연스럽습니다.

**deterministic 설계를 잘 지켰습니다.**
지표 계산과 scoring이 같은 입력에 대해 같은 결과를 내므로 재현성과 reviewability가 좋습니다. 이 프로젝트의 설명 가능성 목표와 맞습니다.

### 2.2 유효한 개선 포인트

**A1. `pipeline.py`에 책임이 많이 몰려 있습니다.**
한 파일 안에 아래가 함께 있습니다.
- provider 구현체
- benchmark / earnings context merge
- snapshot schema와 builder
- helper 함수
- main pipeline orchestration

이건 기능 오류라기보다 유지보수 비용 문제입니다. 다음 구조 변경 시 아래 정도로 나누는 게 적절합니다.
- `pipeline/providers.py`
- `pipeline/context.py`
- `pipeline/snapshot.py`

**A2. `INDICATOR_SNAPSHOT_KEYS`가 암묵 스키마입니다.**
현재 snapshot allowlist는 47개 필드 tuple로 관리됩니다. 새 지표를 넣을 때 계산 로직과 snapshot allowlist를 둘 다 수정해야 하므로 결합이 숨어 있습니다. 방향은 둘 중 하나가 좋습니다.
- allowlist를 별도 contract로 명시
- 반대로 blocklist 방식으로 뒤집어 기본 포함, 예외 제외

**A3. 시간대가 로직 중앙에서 명시되지 않습니다.**
`generated_at`은 시스템 로컬 timezone에 의존하고, `run_date`는 naive `date` 입니다. 서버가 한국/UTC 환경일 때 뉴욕 거래일과 어긋날 여지가 있습니다. 운영으로 커버할 수도 있지만, 장기적으로는 America/New_York 기준 정규화가 안전합니다.

**A4. Oracle SQL schema 관리가 runtime 경로에 들어 있습니다.**
`_ensure_schema()`가 매 persistence 시도마다 DDL block을 실행합니다. 지금도 동작은 하지만, schema migration과 write path를 분리하는 편이 운영상 더 단순합니다.

**A5. 내부 DTO에 Pydantic이 조금 과합니다.**
`PipelineContext`, `RunArtifacts`는 validation/serialization 필요성이 약한 내부 전파 객체라 dataclass로 내려도 충분합니다. 반면 artifact로 직렬화되는 `ScreenRunResult`, `CandidateResult`, `RunMetadata`는 Pydantic 유지가 자연스럽습니다.

**A6. 점수 타입 의도가 흐립니다.**
ranking 단계는 `int` 점수를 만들지만, 모델 경계에서는 `float`로 승격됩니다. 둘 중 하나로 통일하는 게 좋습니다.

### 2.3 이전 초안에서 수정된 판단

**R1. “README가 존재하지 않는 문서를 참조한다”는 지적은 현재 기준으로 틀립니다.**
현재 문서 구조는 정리돼 있고, `README`는 실제 존재하는 `docs/architecture.md`, `docs/signals.md`, `docs/operations.md`만 가리킵니다.

**R2. 그 지적은 원래 기준으로 삼은 `dc5b4ca` 커밋에도 맞지 않았습니다.**

## 3. Algorithm Review

### 3.1 Hard Filter

현재 hard filter는 목적에 맞습니다.
- `bars_available >= 60`
- `average_volume_20d >= 1_000_000`
- Bollinger lower proximity
- `distance_to_20d_low <= 5.0`
- severe weekly damage 제외

turnaround 후보 발굴이라는 목적과 직접 연결되는 조건들입니다. NASDAQ-100 정적 universe를 전제로 하면 지나치게 느슨하지도, 과도하게 복잡하지도 않습니다.

### 3.2 Score Bucket 설계

**oversold / bottom context / volume**
- 비교적 단순하고 설명 가능성이 높습니다.
- threshold도 직관적으로 읽힙니다.

**reversal bucket**
- 현재 가장 복잡한 버킷입니다.
- candle bonus를 여러 조건으로 누적한 뒤 최종 cap 25를 씌우므로, 베이스 점수가 이미 높을 때는 추가 bonus가 많이 유실됩니다.
- 설계 의도라면 괜찮지만, 아니라면 bucket cap 또는 candle bonus weight를 다시 볼 가치가 있습니다.

**market context**
- weekly trend와 QQQ relative strength를 결합한 구조는 좋습니다.
- 다만 weak trend, weak relative strength, earnings, volatility가 한 후보에 동시에 겹치면 감점이 여러 층으로 누적됩니다.

### 3.3 Overlay Penalties

**earnings penalty**
- `<= 2일`: 강한 penalty
- `<= 5일`: 중간 penalty
- recent earnings: 추가 penalty

이 자체는 합리적입니다. 다만 earnings 데이터가 없는 종목은 penalty 0으로 통과하기 때문에, “데이터 없음”을 얼마나 관대하게 볼지 정책을 더 명시할 필요는 있습니다.

**volatility penalty**
- ATR, daily range, BB width를 함께 보는 방식은 현실적입니다.
- 단순한 threshold 기반이지만, false positive 억제에는 충분히 설명 가능한 구조입니다.

### 3.4 가장 큰 알고리즘 리스크

가장 큰 리스크는 factor 부족이 아니라 **calibration 부재**입니다.
- threshold가 여러 곳에 하드코딩돼 있고
- 100점 구조가 직관적으로 설계되어 있으며
- 과거 데이터에 대한 체계적 검증 루프가 아직 없습니다

따라서 다음 단계는 factor 추가보다 backtest skeleton과 threshold 검증이 더 중요합니다.

## 4. Documentation Review

현재 문서 체계는 이전보다 훨씬 건강합니다.

### 4.1 현재 구조
- `README.md`: 프로젝트 개요, 범위, 빠른 시작, 문서 진입점
- `docs/architecture.md`: 구현 구조, output, persistence
- `docs/signals.md`: 현재 규칙의 실제 동작
- `docs/operations.md`: 운영 명령과 환경변수

이 정도면 “현재 구현 기준 문서”로 충분히 읽힙니다.

### 4.2 보완하면 좋은 점

**D1. `signals.md`에 총점 수식을 짧게 명시하면 좋습니다.**
현재 bucket 설명은 잘 되어 있지만, 최종 점수가
`sum(subscores) - earnings_penalty - volatility_penalty`
구조라는 점을 문서에 한 줄 더 써두면 읽는 사람이 더 빠르게 이해할 수 있습니다.

**D2. timezone 전제를 operations 문서에 더 분명히 적을 수 있습니다.**
현재는 명령과 환경변수는 충분하지만, “어느 거래일 기준으로 run_date를 넣는가”는 운영상 민감한 포인트라 명시 가치가 있습니다.

**D3. 이 리뷰 문서 자체도 시점 의존성을 더 명확히 적는 편이 좋습니다.**
commit hash를 머리에 써두는 방식은 좋고, 이후 문서도 그 방식을 유지하면 됩니다.
