# V1.1 Update Plan: OpenClaw Alert Event Sidecar

## 1. Goal

v1.1의 목표는 기존 NASDAQ turnaround screener를 직접 텔레그램을 보내는 시스템으로 바꾸는 것이 아니라, OpenClaw가 바로 소비할 수 있는 `alert-ready` signal producer로 확장하는 것이다.

우선순위는 아래와 같다.

- `machine-consumable signal source`
- `monitoring / ops feed`
- `human research`

즉 v1.1은 사람이 읽기 좋은 보고서보다, OpenClaw가 안전하게 읽고 후속 알림을 결정할 수 있는 안정된 계약을 추가하는 데 초점을 둔다.

## 2. Current Baseline And Freeze Rule

현재 v1.0의 기준 범위는 아래와 같다.

- daily screening
- staged intraday collection
- file artifact 생성
- optional Oracle SQL persistence
- backtest skeleton

기존 진입점과 핵심 artifact 계약은 유지한다.

- `python -m screener.cli.main run`
- `python -m screener.cli.main collect-window`
- `python -m screener.cli.main backtest`
- `python -m screener.cli.main init-oracle-schema`
- `output/daily/*/daily-report.json`
- `output/daily/*/run-metadata.json`
- `output/intraday/*/collection-metadata.json`
- `output/intraday/*/collected-quotes.json`

v1.1은 additive 변경만 허용한다. 기존 artifact를 깨거나, OpenClaw가 기존 report를 해석해야만 알림을 보낼 수 있는 구조로 바꾸지 않는다.

## 3. Product Boundary

v1.1의 경계는 아래와 같다.

- 이 저장소는 텔레그램 전송을 직접 수행하지 않는다.
- 이 저장소는 `alert-events.json` sidecar를 생성한다.
- OpenClaw는 `alert-events.json` 을 읽고 텔레그램 발송 여부를 결정한다.
- intraday 경로는 `provisional` alert를 만든다.
- daily 경로는 `final` alert를 만든다.
- high-priority ticker는 개별 event로 내보내고, 나머지는 digest event로 묶는다.
- 향후 Oracle consumer로 확장할 수 있도록 file-first 계약을 명확히 정의한다.

이 경계는 `Telegram Bot API integration` 과 `delivery orchestration` 을 OpenClaw에 남기고, 이 저장소는 `signal evaluation + alert contract generation` 에 집중하게 만든다.

## 4. Why Alert Event Sidecar

이번 버전의 선택지는 세 가지였다.

- 기존 report를 OpenClaw가 직접 해석하는 minimal overlay
- 별도 `alert-events.json` 을 생성하는 alert event sidecar
- 결과 모델 자체를 알림 도메인 중심으로 재편하는 구조 변경

선택은 `Alert Event Sidecar` 이다.

선정 이유는 아래와 같다.

- v1.0 report 계약을 프리징하기 쉽다.
- OpenClaw가 score 공식과 내부 reason 조합을 몰라도 된다.
- 알림 소비 계약과 raw report를 분리할 수 있다.
- file consumer에서 Oracle consumer로 옮길 때 schema를 재사용하기 쉽다.
- 지금 시점에서 필요한 범위를 넘는 대규모 리팩터를 피할 수 있다.

## 5. Output Contract

### 5.1 New Files

새 artifact는 기존 output 구조 옆에 추가한다.

- daily run-scoped: `output/daily/YYYY-MM-DD/alert-events.json`
- daily stable entrypoint: `output/daily/latest/alert-events.json`
- intraday run-scoped: `output/intraday/YYYY-MM-DD/window-XX-of-YY/run-.../alert-events.json`
- intraday stable entrypoint: `output/intraday/YYYY-MM-DD/latest-alert-events.json`

OpenClaw는 stable entrypoint만 읽는 것을 기본 정책으로 둔다.

- daily consumer path: `output/daily/latest/alert-events.json`
- intraday consumer path: `output/intraday/<NY_DATE>/latest-alert-events.json`

run-scoped artifact는 감사, 디버깅, 회귀 비교 용도로 남긴다.

### 5.2 Top-Level Schema

`alert-events.json` top-level은 아래 필드를 가진다.

```json
{
  "schema_version": 1,
  "delivery_contract": "openclaw-alert-events-v1",
  "run_date": "2026-04-22",
  "generated_at": "2026-04-22T20:05:00Z",
  "run_mode": "daily",
  "phase": "final",
  "source": {
    "artifact_directory": "output/daily/2026-04-22",
    "report_path": "output/daily/2026-04-22/daily-report.json",
    "metadata_path": "output/daily/2026-04-22/run-metadata.json",
    "window_index": null,
    "window_number": null,
    "total_windows": null
  },
  "summary": {
    "eligible_candidate_count": 7,
    "individual_event_count": 2,
    "digest_event_count": 1,
    "suppressed_candidate_count": 4,
    "quality_gate": "pass"
  },
  "events": []
}
```

의도는 아래와 같다.

- `schema_version`: sidecar schema 버전
- `delivery_contract`: OpenClaw consumer가 읽어야 할 논리 계약명
- `phase`: `provisional` 또는 `final`
- `source`: 원본 report와 metadata 위치
- `summary`: 알림 규모와 운영 상태 요약

### 5.3 Event Types

event는 두 종류로 제한한다.

- `ticker_alert`
- `digest_alert`

공통 필드는 아래를 포함한다.

- `event_type`
- `phase`
- `delivery_mode`
- `delivery_priority`
- `severity`
- `dedupe_key`
- `group_key`
- `change_status`
- `change_reason_codes`
- `message_summary`

값 의미는 아래로 고정한다.

- `delivery_mode`: `single` 또는 `digest`
- `delivery_priority`: `high` 또는 `normal`
- `severity`: v1.1에서는 `info` 또는 `warning`

초기 severity 규칙은 아래를 따른다.

- `digest_alert` 는 기본 `info`
- `ticker_alert` 는 기본 `warning`
- 단 `quality_gate == warn` 이면 모든 event에 `warning` 을 유지한다.

`ticker_alert` 는 아래 상세를 포함한다.

- `ticker`
- `name`
- `score`
- `rank`
- `subscores`
- `reasons`
- `risks`
- `indicator_snapshot`
- `headline_reason`
- `headline_risk`
- `source_candidate_ref`

`digest_alert` 는 아래 축약 멤버를 포함한다.

- `member_count`
- `members`

각 member는 아래 필드만 가진다.

- `ticker`
- `name`
- `score`
- `rank`
- `headline_reason`
- `headline_risk`
- `change_status`

## 6. Alert Timing

### 6.1 Provisional

각 intraday run 종료 후 provisional alert를 계산한다.

중요한 제약은 아래와 같다.

- `collected-quotes.json` 만 읽어 바로 alert를 만들지 않는다.
- 최신 staged quote를 기존 daily history와 merge한다.
- 기존 daily scoring pipeline과 같은 규칙으로 provisional candidate를 다시 계산한다.

이렇게 해야 provisional과 final을 같은 scoring 언어로 비교할 수 있다.

### 6.2 Final

장 마감 후 daily run 결과로 final alert를 계산한다.

final alert는 `daily-report.json` / `run-metadata.json` 을 작성한 뒤 생성한다.

## 7. Quality Gate

모든 run은 event 생성 전에 quality gate를 평가한다.

- `pass`
- `warn`
- `block`

초기 기준은 아래로 둔다.

`block`

- `failed_ticker_count > 20`
- 또는 `bars_nonempty_count < 80`
- 또는 `latest_bar_date_mismatch_count > 10`

`warn`

- `failed_ticker_count > 5`
- 또는 `latest_bar_date_mismatch_count > 0`
- 또는 `insufficient_history_count > 5`

그 외는 `pass`

동작 원칙은 아래와 같다.

- `pass`, `warn`: event 생성 허용
- `block`: event 생성 차단, 대신 top-level summary에 상태를 남김

OpenClaw는 `quality_gate == block` 인 artifact를 발송 대상으로 쓰지 않는다.

### 7.1 Final Run Quality Inputs

daily final 경로는 `ScreenRunResult.metadata` 를 기준으로 quality gate를 계산한다.

- `failed_ticker_count`
- `bars_nonempty_count`
- `latest_bar_date_mismatch_count`
- `insufficient_history_count`

### 7.2 Provisional Run Quality Inputs

intraday provisional 경로는 intraday collection 결과와 provisional screening 결과를 함께 본다.

초기 기준은 아래를 따른다.

`block`

- `collected_count < 20`
- 또는 `failed_count > 20`
- 또는 `skipped_due_to_credit_exhaustion_count > 0`

`warn`

- `collected_count < 60`
- 또는 `failed_count > 5`

그 외는 `pass`

이 규칙의 의도는 intraday provisional alert가 같은 날의 전체 universe를 충분히 커버하지 못했을 때 OpenClaw가 과신하지 않도록 만드는 것이다.

## 8. Trigger And Suppression Rules

### 8.1 Candidate Tiers

candidate는 아래 세 단계로 분류한다.

- `suppressed`
- `digest-eligible`
- `single-eligible`

초기 기본 규칙은 아래를 따른다.

`suppressed`

- `score < 45`
- 또는 `reasons` 가 2개 미만
- 또는 `indicator_snapshot.earnings_penalty >= 8`
- 또는 `indicator_snapshot.volatility_penalty >= 4` 이면서 `score < 60`

`digest-eligible`

- `score >= 45`
- `reasons` 2개 이상
- run 내 상위 10개 이내

`single-eligible`

- `score >= 60`
- run 내 상위 5개 이내
- reversal 계열 reason 최소 1개 포함
- oversold 또는 local-bottom 계열 reason 최소 1개 포함
- `change_status` 가 `new`, `upgraded`, `material_change` 중 하나

### 8.2 Hybrid Delivery

같은 run에서 single 조건을 만족한 ticker는 아래처럼 처리한다.

- `ticker_alert` 1건 생성
- digest members 에서는 제외

digest는 나머지 `digest-eligible` candidate만 묶는다.

- intraday run당 최대 1건
- daily final run당 최대 1건
- member 수는 최대 10개

## 9. Change Detection And Resend Rule

### 9.1 Change Status

같은 거래일 안에서 직전 emitted 상태와 비교해 아래를 계산한다.

- `new`
- `upgraded`
- `material_change`
- `unchanged`

정의는 아래와 같다.

- `new`: 오늘 처음 alert-eligible 상태가 됨
- `upgraded`: `digest -> single` 또는 더 높은 delivery priority로 승격
- `material_change`: tier는 같지만 의미 있는 차이가 생김
- `unchanged`: 다시 계산했지만 재전송 가치가 없음

### 9.2 Material Change

아래 중 하나라도 참이면 `material_change` 로 본다.

- `score_delta >= 5`
- `rank_delta >= 2`
- `headline_reason` 변경
- `headline_risk` 변경
- `earnings_penalty` 변경
- `volatility_penalty` 변경
- `phase` 가 `provisional -> final` 로 바뀌면서 score 또는 headline이 함께 변경

### 9.3 Final Resend

장중 provisional single alert가 이미 발행된 ticker라도 final run에서는 아래를 모두 만족할 때만 다시 single alert를 생성한다.

- final 시점에도 `single-eligible`
- `change_status != unchanged`

즉 final 재알림은 "장 마감" 자체가 아니라 "의미 있는 변화"가 있어야 발생한다.

## 10. Internal State And Dedupe

### 10.1 Internal State File

`changed-only` 판정을 위해 내부 상태 파일을 둔다.

- path: `output/alerts/YYYY-MM-DD/alert-state.json`

이 파일은 OpenClaw 소비 계약이 아니라, 이 저장소 내부의 alert sidecar 생성 상태를 위한 것이다.

### 10.2 Stored Fields

ticker별 상태는 아래를 저장한다.

- `last_delivery_tier`
- `last_material_signature`
- `last_phase`
- `last_emitted_at`
- `last_dedupe_key`

digest 상태는 아래를 저장한다.

- `last_digest_signature`
- `last_digest_dedupe_key`

### 10.3 Signature And Dedupe Key

`dedupe_key` 는 wall-clock timestamp가 아니라 material signature 기반으로 만든다.

예시:

- `nasdaq-turnaround:2026-04-22:AAPL:single:8f3a91c2`
- `nasdaq-turnaround:2026-04-22:provisional:digest:42ab77de`

이 구조의 목적은 아래와 같다.

- 같은 내용이면 같은 dedupe key가 다시 나온다.
- producer state commit이 지연되더라도 OpenClaw가 재전송을 막을 수 있다.
- producer와 consumer가 중복 방지를 이중으로 수행할 수 있다.

## 11. Write Order And Safe File Update

sidecar 생성 시 파일 쓰기 순서는 아래를 따른다.

1. 기존 `alert-state.json` 읽기
2. 현재 run의 event 계산
3. run-scoped `alert-events.json` 을 임시 파일에 쓰고 atomic rename
4. stable entrypoint 갱신
5. 새 `alert-state.json` 을 임시 파일에 쓰고 atomic rename

원칙은 아래다.

- state가 event보다 먼저 진전되면 안 된다.
- 중간 장애 시 최악의 결과는 "이벤트가 한 번 더 보이는 중복" 이어야 한다.
- 중복은 OpenClaw가 `dedupe_key` 로 막는다.

## 12. Failure Semantics

raw report와 alert sidecar의 실패는 같은 의미로 취급하지 않는다.

- raw report/artifact 쓰기는 먼저 수행한다.
- sidecar generation이 실패하면 metadata notes에 가능한 한 실패 사유를 남긴다.
- sidecar generation 실패는 command를 `non-zero exit` 로 종료시킨다.

이 정책의 이유는 아래와 같다.

- v1.0 truth artifact는 남겨야 한다.
- 하지만 v1.1 이후 OpenClaw는 alert contract를 운영 경로로 의존할 수 있다.
- 따라서 alert sidecar 누락을 성공으로 처리하면 운영 위험이 커진다.

## 13. Internal Module Plan

새 기능은 별도 `alerts` 모듈로 추가한다.

- `src/screener/alerts/schema.py`
- `src/screener/alerts/policy.py`
- `src/screener/alerts/state.py`
- `src/screener/alerts/builder.py`
- `src/screener/alerts/writer.py`
- `src/screener/alerts/__init__.py`

책임은 아래처럼 나눈다.

- `schema.py`: Pydantic model과 payload serialization
- `policy.py`: quality gate, tier 판정, change status 계산
- `state.py`: state load/save, signature hash, atomic write
- `builder.py`: `ScreenRunResult` 또는 provisional evaluation 결과에서 event 계산
- `writer.py`: run-scoped path와 stable entrypoint 기록

이 분리는 기존 `pipeline`, `collector`, `reporting`, `storage` 책임을 유지하면서 alert logic을 독립 테스트 가능하게 만든다.

## 14. Invocation Points

### 14.1 Daily

daily run 경로에서 아래 순서로 호출한다.

1. screening 수행
2. markdown / json / metadata artifact 기록
3. final alert sidecar 생성
4. 필요 시 Oracle persistence 수행

### 14.2 Intraday

intraday run 경로에서 아래 순서로 호출한다.

1. quote collection 수행
2. `collection-metadata.json` / `collected-quotes.json` 기록
3. latest staged quote 기반 provisional screening 수행
4. provisional alert sidecar 생성
5. 필요 시 Oracle persistence 수행

intraday provisional screening은 기존 daily scoring 규칙을 재사용해야 한다.

## 15. Testing Strategy

테스트는 네 층으로 나눈다.

### 15.1 Unit Tests

- quality gate 판정
- candidate tier 분류
- `change_status` 계산
- material signature hash 생성
- dedupe key 생성

### 15.2 Schema Tests

- top-level schema 필드 확인
- `ticker_alert` / `digest_alert` payload shape 확인
- stable path naming 확인

### 15.3 Integration Tests

- daily run 후 `alert-events.json` 생성 확인
- `output/daily/latest/alert-events.json` 갱신 확인
- intraday run 후 run-scoped provisional sidecar 생성 확인
- `output/intraday/<date>/latest-alert-events.json` 갱신 확인

### 15.4 State Transition Tests

같은 ticker가 아래 순서를 거칠 때 event 수와 `change_status` 가 기대대로 변하는지 확인한다.

- `suppressed -> digest`
- `digest -> single`
- `single -> unchanged`
- `provisional -> final`

추가로 fixture 기반 golden test를 둬서 `alert-events.json` 의 의미 있는 변경이 diff로 드러나게 한다.

## 16. Rollout Plan

도입은 세 단계로 나눈다.

### 16.1 Producer Only

- 이 저장소가 sidecar만 생성한다.
- OpenClaw는 아직 전송하지 않는다.

### 16.2 Consumer Shadow Mode

- OpenClaw가 `alert-events.json` 을 읽는다.
- quality gate, dedupe key, delivery mode를 해석한다.
- 실제 텔레그램 전송 대신 로그만 남긴다.

### 16.3 Send Enabled

- Telegram 전송 활성화
- 초기에는 digest와 single 모두 켠다.
- OpenClaw의 dedupe 상태 저장이 안정적인지 먼저 확인한다.

## 17. OpenClaw Consumer Contract

OpenClaw는 아래 규칙만 따르면 된다.

- stable entrypoint만 읽는다.
- `schema_version` 과 `delivery_contract` 를 확인한다.
- `summary.quality_gate` 가 `block` 이면 발송하지 않는다.
- 자신이 이미 보낸 `dedupe_key` 는 별도 저장한다.
- 텔레그램 포맷팅은 OpenClaw가 담당한다.

즉 이 저장소는 "무엇을 보낼 가치가 있는지"를 결정하고, OpenClaw는 "어떻게 보낼지"를 결정한다.

## 18. Scope Cut

### Included In V1.1

- file-based alert sidecar
- provisional / final phase
- hybrid single / digest
- local alert state
- stable consumer entrypoints
- changed-only resend

### Excluded From V1.1

- Telegram direct send
- Oracle-based alert consumer
- portfolio queue
- 매수 / 매도 실행 엔진
- UI dashboard
- 사용자 설정형 alert rule editor

## 19. Success Criteria

v1.1이 성공했다고 볼 기준은 아래와 같다.

- 기존 v1.0 report consumer가 깨지지 않는다.
- OpenClaw가 raw report 해석 없이 `alert-events.json` 만으로 알림 결정을 할 수 있다.
- 같은 거래일 내 중복 텔레그램이 `dedupe_key` 기준으로 억제된다.
- provisional과 final 사이의 재알림은 의미 있는 변화가 있을 때만 발생한다.
- quality 문제가 클 때 downstream 발송을 차단할 수 있다.

## 20. Implementation Notes For Next Step

다음 단계의 구현 계획에서는 아래 순서를 우선 고려한다.

1. alert schema와 state model 도입
2. daily final sidecar 먼저 구현
3. daily integration test 추가
4. intraday provisional evaluation 추가
5. intraday state transition test 추가
6. 운영 문서 업데이트

이 순서는 daily 경로에서 file-based contract를 먼저 안정화한 뒤, provisional 경로와 OpenClaw shadow mode를 붙이기 위한 것이다.
